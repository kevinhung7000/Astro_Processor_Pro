## English

Candidate improvements for the next release, in suggested priority order. Not a strict commitment — just current direction, same as `ROADMAP.md`.

### High priority

- **~~Batch / export timing~~ ✅ Done (v1.2.0)** — the live preview already shows elapsed time per run (added in v1.1.0), but batch processing and full-resolution export still give no time feedback at all. Someone running batch on 40 images has no idea whether it'll take 2 minutes or 20. Add per-file timing to the batch progress log, and a total elapsed time to the export success message.
  - *Effort*: low — same `time.perf_counter()` pattern already used for preview, just applied to `batch_process_fn` and `export_fn`.
  - > **Code review note**: Confirmed straightforward. `batch_process_fn` is already a generator with `yield` per file, and `export_fn` has a single clear success-message return point — both are natural insertion points for `time.perf_counter()`. Effort estimate holds as-is; this is the best cost/value item on the list.
  - > **Implementation note (v1.2.0)**: Shipped. `export_fn` now reports total elapsed time on both success and failure. `batch_process_fn` now logs per-file elapsed time in each progress line, and — as a natural small extension beyond the original scope — also shows a running ETA (elapsed so far / estimated time remaining, based on the average per-file time observed so far). Flagging the ETA line as a scope addition in case it's more than wanted on screen; easy to drop if it makes the progress log too busy.

- **~~Config export/import — parameterized filename~~ ✅ Done (v1.2.0)** *(new, split out from the snapshot item below)*
  - `export_config_fn` currently always writes to a fixed filename `astro_config.json`, overwriting any previous export. This needs to change to a parameterized filename before named snapshot-to-file saving (below) can work without one snapshot silently overwriting another.
  - *Effort*: low — small, self-contained change; worth doing together with the snapshot work since it's a prerequisite for it.
  - > **Implementation note (v1.2.0)**: Shipped together with the snapshot work below. Added an optional "Export Filename" textbox; if left blank, falls back to the original `astro_config.json` default (unsanitized user input is filtered to safe characters — alphanumeric, underscore, hyphen, and CJK — to avoid path-traversal or invalid-filename issues).

### Medium priority

- **~~Named, savable snapshots~~ ✅ Done (v1.2.0)** — Snapshot A/B/C (v1.1.0) are session-only and unlabeled. Add an optional name field per snapshot, and a "save snapshot to file" / "load snapshot from file" pair (reusing the existing JSON config export/import plumbing) so a favorite setup like "backyard light pollution" survives a page refresh.
  - *Effort*: medium — mostly UI wiring, minimal new logic since JSON export/import already exists.
  - > **Code review note**: Current `save_snapshot_fn` / `load_snapshot_fn` are indeed pure `gr.State`, session-only — matches the description exactly. But this item depends on the fixed-filename issue above being fixed first, otherwise saving a second named snapshot will overwrite the first one on disk. Recommend doing the filename parameterization item together with this one in the same change.
  - > **Implementation note (v1.2.0)**: Shipped. Added a "Target Slot" (A/B/C) selector, an optional "Snapshot Name" field, a "Save Snapshot to File" button, and a "Load Snapshot from File" upload. Save writes a JSON payload (`snapshot_name`, `slot`, `saved_at`, `params`) to a name-derived filename (e.g. `astro_snapshot_backyard.json`); load reads it back into the chosen A/B/C slot's `gr.State` and status label — the user still needs to click "Apply A/B/C" afterward to push it onto the sliders, matching how loading already worked for in-session snapshots. Also made `import_config_fn` and the new snapshot loader tolerant of each other's file format (both a bare params dict and a `{"params": {...}}` wrapper are accepted), so a snapshot file and a plain config export are interchangeable.

- **~~Collapse Presets / Snapshots into accordions~~ ✅ Done (v1.2.0)** *(new, small UI follow-up to the two items above)* — Beginner Presets and Parameter Snapshots were plain, always-open `gr.Group` blocks sitting above the numbered parameter accordions (`1️⃣ Background Gradient Removal`, etc.), taking up permanent vertical space even for users who never touch them after their first session.
  - *Effort*: low — pure UI change, no pipeline logic involved; same `gr.Accordion(..., open=False)` pattern already used everywhere else in the parameter panel.
  - > **Implementation note (v1.2.0)**: Shipped. Both sections are now collapsible accordions — `0️⃣a 🎛️ Beginner Presets` and `0️⃣b 📌 Parameter Snapshots` — collapsed by default (`open=False`), matching the existing numbered accordions rather than sitting permanently expanded. Numbered `0️⃣a`/`0️⃣b` to signal they run before the `1️⃣`+ processing steps. Language switching (zh/en) for both accordion titles and their contents continues to work as before.

- **~~Manual region selection for local stretch~~ ✅ Done (v1.2.0)** — Auto Localized Stretch (v1.1.0) detects textured regions automatically, but if it misses a faint target or over-boosts a noisy patch, there's currently no way to override it. Add an optional manual mask: let the user draw/select a region (e.g. via the crop-preview coordinates already used for local preview) that's always boosted regardless of what auto-detection finds.
  - *Effort*: ~~medium-high~~ **high** — needs a way to capture a user-drawn region and merge it with the existing per-pixel mask; the local-preview crop selector is a natural starting point.
  - > **Code review note**: `local_x_pct` / `local_y_pct` / `local_crop_px` do exist and are a reasonable starting point, but today they only drive a **preview crop overlay** (`_update_overview_only`), not the pipeline's actual pixel mask — these are two separate data paths. Turning a user-selected rectangle into something that merges with the existing auto-detected per-pixel mask requires: (1) converting percentage-based preview coordinates to full-resolution pixel coordinates, (2) feathering the edges of the manual region so it doesn't create a hard, visible boundary, and (3) deciding the merge policy with the auto mask (union? user override wins? adjustable blend weight?). The UI part is simple; the masking/merge logic is the real work. Bumping effort from medium-high to **high**.
  - > **Implementation note (v1.2.0)**: Shipped, in `3️⃣b Auto Localized Stretch`, below the auto-detection controls. Ended up using dedicated percentage-based sliders (X/Y center, width/height, all as % of image dimensions) rather than reusing `local_x_pct`/`local_y_pct`/`local_crop_px` directly — those drive the separate ROI-preview crop tool (a different data path, per the code review note above), and percentage-based sizing (rather than the ROI tool's absolute `crop_px`) means the region works correctly at any resolution — full export, thumbnail preview, or the ROI-preview crop — without needing `preview_scale` conversion. Resolved the three open design questions as: (1) percent-to-pixel conversion happens fresh against whatever image the pipeline is currently running on; (2) edge feathering via `gaussian_filter`, with a "Edge Feather (%)" slider sized relative to the region's shorter side; (3) merge policy is **union with adjustable weight** — `final_mask = max(auto_mask, manual_mask)`, where the manual region's mask value is set by a "Region Boost Weight" slider (0–1) rather than always forcing 1.0, so it can add a gentle nudge or a full override. The manual region also works independently of the auto-detection toggle (auto can stay off and the manual region alone still boosts). No dedicated visual overlay was added for the manual rectangle specifically — the existing "Auto Localized Stretch mask" preview already shows the *merged* mask (auto + manual) as a red highlight with a yellow contour, so the manual region's actual coverage is visible there without extra UI.
  - > **Follow-up (v1.2.0)**: Added a "Region Shape" toggle (rectangle / ellipse) next to the enable checkbox. Rectangle keeps the original hard-edged-box-then-feather behavior; ellipse computes the region as a normalized-distance mask (`((x-x0)/(rw/2))² + ((y-y0)/(rh/2))² ≤ 1`) before the same `gaussian_filter` feathering is applied, so both shapes share identical edge-softening behavior. No separate "circle" option was added — setting width % = height % on the ellipse shape already gives a perfect circle, so a third choice would have been redundant. `build_manual_target_mask()` and `apply_local_target_boost()` both take the new `shape` parameter (default `"rectangle"`, so existing configs/snapshots without the field keep working unchanged); the merged-mask preview overlay needed no changes since it already visualizes whatever the merged per-pixel mask happens to contain.

### Low priority

- **ML-based denoising** — quality mode (v1.1.0) is Non-local Means, not a learned denoiser. A real ML option (e.g. a small pretrained model) would help with noisier subs, but has real costs: a noticeably larger installer, a new inference dependency, and slow CPU-only performance for users without a GPU. Worth scoping only after the above are done.
  - *Effort*: high — model selection, packaging size/dependency tradeoffs, and a CPU fallback story all need deciding before writing any code.
  - > **Code review note**: `AstroProcessorPro_Setup.iss` already notes the installer bundles `torch` for GPU-accelerated background estimation ("因為含 torch，體積會很大"). **If** the shipped build already includes torch, the "new inference dependency" cost in this item is overstated — the incremental cost would mainly be model weight file size, not a whole new dependency. This only applies to builds that ship with torch included; a slimmer, torch-free build would still face the full cost as originally described. Worth confirming which build variant is actually distributed before re-scoping this item's priority.
  - > **Research note**: Surveyed the available ready-made ML denoisers and their licensing before scoping further. **DeepSNR** (free, built on the MIT-licensed NAFNet architecture, by the same author as StarNet) and **Seti Astro Cosmic Clarity** (free) exist as standalone/PixInsight tools; **RC-Astro NoiseXTerminator** is a paid (~$49.95+), closed-source, per-license-key product that recently also shipped a CLI. None of these can simply be copied into our own installer: DeepSNR/Cosmic Clarity are distributed for end users to install themselves rather than for third-party redistribution, the original StarNet weights are explicitly licensed non-commercial, and NoiseXTerminator requires each user to hold their own paid, activated license — there's no way to ship it pre-activated. Two realistic paths forward: (a) **external-tool integration** — detect an already-installed DeepSNR/RC-Astro CLI on the user's machine and shell out to it (the same approach Siril uses for RC-Astro), which sidesteps redistribution entirely since the user handles their own install and licensing; (b) **train our own small model** on NAFNet or a similarly MIT-licensed architecture using our own noisy/clean astro pairs, so the resulting weights are ours to bundle freely — this is literally the same path DeepSNR's author took. Directly bundling someone else's weights without written permission is the one option that's off the table. Given the existing `torch` dependency noted above, path (b)'s incremental installer cost would mainly be weight file size, but it still needs a training dataset and iteration to reach usable quality, so effort stays **high**. Related aside: **ML-based star removal faces the identical licensing shape** — StarNet/StarNet2 (free, but non-commercial-licensed weights) and RC-Astro StarXTerminator (paid, per-license-key) are the mainstream tools, so if a future item ever proposes upgrading beyond the current classical shrink/remove star removal, the same two paths (external CLI integration or train-our-own) would apply.
  - > **~~Path (a): external-tool integration~~ ✅ Done (v1.2.0)** — A closer look at path (a)'s actual implementation, covering both denoising and star removal since they'd share the same architecture.
    - **The data path is genuinely new work, not an extension of existing functions**: `denoise(img8, ...)` and `process_stars(mode, img01, ...)` are both pure in-memory-array-in, in-memory-array-out function calls today; DeepSNR/Cosmic Clarity/NoiseXTerminator/StarXTerminator are all standalone executables that work file-in, file-out instead. This means adding a translation layer: `img8` (or the full-resolution export buffer) → write to a temp file (16-bit TIFF recommended, rather than the pipeline's internal 8-bit, to avoid losing dynamic range before the denoise/star step even runs) → `subprocess` call to the external CLI with an output path → read the result file back → convert back to `img01`/`img8` for the rest of the pipeline. This translation layer (temp-file lifecycle, bit-depth/color-space conversion, cleanup) is new work, not something that piggybacks on `denoise()`/`process_stars()`'s existing logic.
    - **Detection should be manual configuration, not an install scan**: scanning the whole disk at startup to guess install locations is fragile and behaves inconsistently across platforms. The more practical approach: let the user point to the executable once in settings, persisted through the existing `export_config_fn`/`import_config_fn` JSON config plumbing; on startup or when switching to this mode, call the executable's `--version` (or equivalent) to confirm it's still valid, and show "tool not detected" in the UI rather than letting the pipeline fail outright.
    - **UI integration point**: `denoise_mode` is currently `gr.Radio(["fast", "quality"], ...)` — a third option like `"external"` can be added; `star_mode` is currently `gr.Radio(["none","shrink","remove"], ...)`, same idea. Selecting it would reveal "external tool path" and "extra arguments" fields within the same accordion, following the same conditional-field pattern `toggles_for_change` already uses elsewhere — no new UI mechanism needed.
    - **Naturally compatible with the v1.2.0 timing/ETA work above**: an external-tool call (subprocess launch + temp file I/O) is inherently slower than the current fast/quality modes, so it can reuse the `time.perf_counter()` instrumentation and ETA estimate already being added to `batch_process_fn`/`export_fn` rather than needing its own feedback mechanism. One wrinkle: the first image's external-tool run will likely take noticeably longer than subsequent ones (process startup, model loading), so the ETA's running-average logic should probably discard that first sample, or flag it as "first-file time excluded from estimate."
    - **Error handling and degradation**: the `subprocess` call needs a timeout plus try/except — missing executable, non-zero exit code, timeout, or a missing/malformed output file all need to be caught. On failure, the recommended default is to **skip that step entirely (equivalent to the mode being off) and show a clear error in that file's progress/export message**, rather than aborting the whole batch or silently falling back to fast/quality (which would make the user think the external tool actually ran).
    - **Licensing boundary needs to be explicit in the UI**: the app itself never touches or redistributes any third-party model weights or installers — it only calls an executable the user installed and licensed themselves (NoiseXTerminator/StarXTerminator require the user's own paid, activated license key; DeepSNR/Cosmic Clarity/StarNet are user-downloaded). A short disclaimer next to the "external tool path" field, making clear licensing responsibility sits with the user, would avoid any impression that this is a built-in or pre-licensed feature.
    - **A performance consideration**: background estimation already uses `torch` (CUDA/DirectML) GPU resources; if the external tool also uses the GPU, running both at once risks contention or OOM. In batch mode, the external-tool call and local GPU-based pipeline steps should probably be serialized rather than run concurrently.
    - **Revised effort estimate**: the glue code for "shell out to an external program" (`subprocess` + temp-file conversion + UI branching + error handling) is moderate effort on its own — well below the "high" estimate for training our own model. The real risk is long-term maintenance, not one-time development: each external tool has its own CLI arguments, input/output formats, and color-space conventions, and future tool versions may change their interface. **Recommendation**: if path (a) is pursued, integrate one tool first (RC-Astro's CLI is a reasonable first target since it covers both NoiseXTerminator denoising and StarXTerminator star removal with a shared interface design), validate the full detect → call → read-back → error-handling flow, then expand to other tools only if it proves worthwhile — rather than building support for all of them at once.
    - > **Implementation note (v1.2.0)**: Shipped. Added a shared `run_external_image_tool()` helper (img01 → 16-bit TIFF temp file → `subprocess` call to the external CLI → read result back → convert back to `img01`), used by a new `mode="external"` on both `denoise()` and `process_stars()`. Supports either `{input}`/`{output}` placeholders in the extra-arguments field or the generic "executable input_file output_file" convention when no placeholder is present. Failures (missing path, missing executable, timeout, non-zero exit code, missing output file, unreadable output) are caught and the step is skipped — falls back to the original image rather than aborting the pipeline or silently substituting fast/quality. New fields (`denoise_ext_path/args`, `star_ext_path/args`) were appended to the end of `PARAM_NAMES`/`DEFAULTS`/`PARAM_COMPONENTS` so existing index-based lookups (e.g. `DEFAULTS[24]`/`DEFAULTS[43]`) are unaffected; config export/import, snapshots, presets, and Reset all pick the new fields up automatically since they already iterate `PARAM_NAMES` generically. One packaging note beyond the original scope: this adds **no new dependency** — `tifffile` was already bundled for reading TIFF source files, so the "external tool integration" feature itself doesn't grow the installer. Tested against the four expected failure/success paths (unset path, missing executable, tool runs but produces no output, successful round-trip) and, separately, against a real DeepSNR CLI build — which confirmed one thing the plan hadn't flagged: DeepSNR's CLI takes named flags (`-i`/`--input`, `-o`/`--output`), not positional arguments, so the generic "executable input_file output_file" fallback doesn't work for it — the extra-arguments field needs `--input {input} --output {output}` explicitly. Still open: only DeepSNR has been verified end-to-end so far; NoiseXTerminator/StarXTerminator CLI syntax remains unverified, and live-preview performance when `mode="external"` is selected still depends entirely on the external tool's own speed, per the known limitations noted when this was first drafted.

### 💬 Feedback

As with the main roadmap, this is a direction, not a commitment — open to reordering based on what actually turns out to matter for real usage.

### 低優先

- **ML 降噪** — quality 模式（v1.1.0）用的是 Non-local Means，並非學習式降噪器。真正的 ML 選項（例如小型預訓練模型）對雜訊較重的疊圖會有幫助，但代價也真實存在：安裝檔會明顯變大、多一個推論相依套件、沒有 GPU 的使用者純 CPU 跑會很慢。建議等前面幾項做完後再評估規劃。
  - *工作量*：高 — 模型選型、封裝大小/相依套件的取捨、以及無 GPU 時的退路方案，這些都得先決定清楚才適合動手寫程式。
  - > **複核備註**：`AstroProcessorPro_Setup.iss` 裡已註明安裝檔為了 GPU 背景估計加速而包含 `torch`（「因為含 torch，體積會很大」）。**如果**目前實際出貨的版本本來就含 torch，這項「多一個推論相依套件」的代價其實被高估了——增量成本主要只剩模型權重檔案大小，而非全新的相依套件。但這個結論只適用於「含 torch」的打包版本；若出貨的是不含 torch 的精簡版，原本描述的完整代價仍然成立。建議先確認實際出貨版本，再決定是否要調整這項的優先度評估。
  - > **研究筆記**：進一步規劃前，先調查了現成 ML 降噪工具與各自的授權狀態。**DeepSNR**（免費，架構是 MIT 授權的 NAFNet，作者跟 StarNet 是同一人）和 **Seti Astro Cosmic Clarity**（免費）都是獨立軟體／PixInsight 外掛；**RC-Astro NoiseXTerminator** 是付費（約 $49.95 起）、閉源、需個人授權金鑰啟用的產品，最近也推出了 CLI 版本。但這些都**不能直接複製進我們自己的安裝檔**：DeepSNR／Cosmic Clarity 是給終端使用者自行下載安裝用的，並非授權給第三方軟體重新分發；原版 StarNet 的權重明確寫著禁止商用；NoiseXTerminator 則要求每個使用者持有自己付費啟用的授權，沒辦法預先啟用後直接出貨。實際可行的路線有兩條：(a) **外部工具介接**——偵測使用者電腦上是否已安裝 DeepSNR／RC-Astro CLI，把處理工作丟給它執行（跟 Siril 整合 RC-Astro 的做法一樣），這樣完全不涉及重新散布別人的東西，授權責任在使用者自己身上；(b) **自己訓練一個小模型**——用 NAFNet 或類似 MIT 授權的架構，拿自己收集的雜訊/乾淨天文影像配對來訓練，訓練出來的權重完全歸自己所有，可以放心內建——這其實就是 DeepSNR 作者當初走的同一條路。唯一真正不能做的，是沒有書面授權就直接把別人的權重塞進安裝檔。考量到前面複核備註提到的既有 `torch` 依賴，路線 (b) 的增量安裝檔成本主要只是權重檔案大小，但仍然需要訓練資料集與反覆調校才能達到堪用品質，工作量估計維持**高**。順帶一提：**ML 去星面臨完全相同的授權處境**——StarNet／StarNet2（免費，但權重授權禁止商用）和 RC-Astro StarXTerminator（付費、需個人授權金鑰）是主流的 ML 去星工具，如果之後有項目想把現有的傳統 shrink/remove 去星升級成 ML 版本，同樣是那兩條路線（外部 CLI 介接，或自己訓練）可選。
  - > **~~方案 A（外部工具介接）~~ ✅ 已完成（v1.2.0）**：針對上面路線 (a) 做進一步的落地可行性評估，降噪與去星共用同一套架構，可以一起規劃。
    - **資料流是新增的一層，不是現有函式的參數延伸**：`denoise(img8, ...)` 和 `process_stars(mode, img01, ...)` 目前都是「記憶體陣列進、記憶體陣列出」的純函式呼叫；DeepSNR／Cosmic Clarity／NoiseXTerminator／StarXTerminator 這類外部工具則都是獨立執行檔，走的是「檔案進、檔案出」。因此需要新增一段轉接邏輯：`img8`（或全解析度匯出時的原始緩衝）→ 寫成暫存檔（建議 16-bit TIFF，而非目前 pipeline 內部用的 8-bit，避免降噪/去星前先損失動態範圍）→ `subprocess` 呼叫外部 CLI 並帶入輸出路徑 → 讀回結果檔 → 轉回 `img01`/`img8` 供後續步驟接手。這一層轉接（暫存檔管理、色彩空間與位元深度轉換、清除暫存檔）是全新工作量，不能算進 `denoise()`/`process_stars()` 既有邏輯的延伸。
    - **偵測方式建議走手動設定，而非自動掃描**：與其在啟動時掃描全機碟猜測安裝路徑（容易誤判、跨平台行為不一致），更務實的做法是在設定裡讓使用者手動指定一次執行檔路徑，沿用 `export_config_fn`／`import_config_fn` 既有的 JSON 設定檔機制存起來；下次啟動或切換到該模式時，嘗試呼叫該執行檔的 `--version`（或等效參數）驗證仍然有效，驗證失敗就在 UI 提示「未偵測到工具」，而不是讓 pipeline 直接失敗。
    - **UI 整合點**：`denoise_mode` 目前是 `gr.Radio(["fast", "quality"], ...)`，可以加第三個選項如 `"external"`；`star_mode` 目前是 `gr.Radio(["none","shrink","remove"], ...)`，同理加 `"external"`。選到該選項時，在同一個 `6️⃣ 降噪` / 對應去星 accordion 內動態顯示「外部工具執行檔路徑」與「額外參數」兩個輸入欄，行為模式與現有 `toggles_for_change` 已經在做的「依模式切換顯示欄位」一致，不需要新的 UI 機制。
    - **與既有 v1.2.0 耗時/ETA 功能天然相容**：外部工具呼叫（啟動子行程 + 寫讀暫存檔）本來就比目前的 fast/quality 模式慢上不少，剛好可以直接沿用本次已經加到 `batch_process_fn`／`export_fn` 的 `time.perf_counter()` 埋點與 ETA 估算，不需要另外設計耗時回饋機制；只是第一張圖跑外部工具時的耗時可能跟後續張數落差較大（子行程啟動、模型載入等一次性成本），ETA 的「用過去平均推算」邏輯在切到外部模式時，建議跳過第一張的樣本或另外提示「首張耗時不列入預估」。
    - **錯誤處理與降級策略**：`subprocess` 呼叫必須包 timeout + try/except——執行檔不存在、CLI 回傳非 0、逾時、輸出檔案沒有正確產生，都要能被攔截；攔截後預設行為建議是**該步驟直接跳過（等同該模式關閉）並在該張圖的進度/匯出訊息顯示明確錯誤**，而不是讓整個 batch 中斷或用預設 fast/quality 靜默取代（會讓使用者誤以為外部工具確實有跑）。
    - **授權界線需要在 UI 上寫清楚**：App 本身完全不觸碰、不重新散布任何第三方模型權重或安裝檔，只是呼叫使用者電腦上「使用者自行安裝、自行取得授權」的執行檔（NoiseXTerminator／StarXTerminator 需要使用者自己的付費授權金鑰啟用；DeepSNR／Cosmic Clarity／StarNet 則是使用者自行下載）。建議在「外部工具路徑」欄位旁加一行說明文字，清楚寫明授權責任在使用者身上，避免使用者誤會這是我們代為授權或內建的功能。
    - **效能面的額外考量**：目前背景估計已經會用到 `torch`（CUDA / DirectML）GPU 資源，如果外部工具本身也吃 GPU，兩者同時執行可能互搶顯示卡資源導致變慢甚至 OOM；批次模式下建議讓「呼叫外部工具」與「本地 GPU 運算步驟」序列化執行，不要並行呼叫。
    - **工作量重新評估**：單純「呼叫外部程式」的膠水程式碼（`subprocess` + 暫存檔轉換 + UI 分支 + 錯誤處理）本身工作量落在中等，遠低於原本「自行訓練模型」評估的高工作量；但每套外部工具的 CLI 參數、輸入輸出格式、色彩空間慣例都不一樣，且未來版本可能改變介面，長期維護成本才是真正的風險，而非一次性開發工作量。**建議**：如果要走方案 A，第一步先支援其中一套工具（例如 RC-Astro 的 CLI，因為它同時涵蓋 NoiseXTerminator 降噪與 StarXTerminator 去星兩種需求，介面設計可以共用），驗證整個「偵測 → 呼叫 → 讀回 → 錯誤處理」的流程後，再視情況擴充支援其他工具，而不是一次把所有工具都做進去。
    - > **實作備註（v1.2.0）**：已完成。新增共用函式 `run_external_image_tool()`（img01 → 16-bit TIFF 暫存檔 → `subprocess` 呼叫外部 CLI → 讀回結果 → 轉回 img01），`denoise()` 與 `process_stars()` 都新增 `mode="external"` 呼叫它。額外參數欄位支援 `{input}`/`{output}` 佔位符，若沒填佔位符則自動退回「執行檔 輸入檔 輸出檔」的通用慣例。所有失敗情況（路徑未設定、找不到執行檔、逾時、非 0 回傳碼、輸出檔缺失、讀檔失敗）都攔截並讓該步驟直接跳過、回傳原圖，不中斷 pipeline，也不會靜默改用 fast/quality 頂替。新參數（`denoise_ext_path/args`、`star_ext_path/args`）附加在 `PARAM_NAMES`/`DEFAULTS`/`PARAM_COMPONENTS` 尾端，不影響既有 `DEFAULTS[24]`／`DEFAULTS[43]` 這類索引取值；設定檔匯出/匯入、快照、Preset、Reset 因為原本就是通用的 `PARAM_NAMES` 迴圈，自動相容不用額外改。額外補一筆原計畫沒提到的好消息：這項功能**沒有新增任何相依套件**——`tifffile` 原本就因為讀取 TIFF 來源檔而已內建，所以外部工具介接本身不會讓安裝檔變大。測試涵蓋了原本規劃的四種情境（未設定路徑／找不到執行檔／工具沒產生輸出／成功呼叫並讀回結果），也額外拿真正的 DeepSNR CLI 實測過，並確認一件計畫階段沒抓到的細節：DeepSNR 的 CLI 吃的是具名旗標（`-i`/`--input`、`-o`/`--output`），不是位置參數，所以「執行檔 輸入檔 輸出檔」這條通用退路對它不適用——額外參數欄位需要明確填 `--input {input} --output {output}` 才能跑。目前仍待確認：只有 DeepSNR 完整驗證過，NoiseXTerminator／StarXTerminator 的 CLI 語法還沒實機測試；切到 `mode="external"` 時的即時預覽速度仍完全取決於外部工具本身，跟最初草擬時記錄的已知限制一致。
